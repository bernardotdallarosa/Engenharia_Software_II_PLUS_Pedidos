2.2 MS Auth eleito — detalhes técnicos

  Backend:  Python 3.12, FastAPI, PostgreSQL, SQLAlchemy, JWT
  Frontend: React 18, TypeScript, Vite, MUI, Module Federation

  Roles no JWT (claim "role"):
    admin
    vendedor
  (minúsculas, em português — distintas de ADMIN/STAFF/MANAGER)

  Claims relevantes no access token:
    sub      — email do utilizador
    user_id  — identificador numérico
    role     — "admin" | "vendedor"

  Endpoints (prefixo /auth):
    POST /register, /login, /refresh, /logout
    GET  /me, /users
    POST /promote/{user_id}

  Porta local: 3001
  Resposta de login: { access_token, refresh_token, token_type: "bearer" }

  Utilizador inicial (seed da BD, criado no arranque do plus-ms-auth):
    email:    admindev@admin.com
    password: Senha123
    role:     admin

  Registo via API (POST /auth/register): novos utilizadores recebem role vendedor.
  Regras de senha: 6–10 caracteres, maiúscula, minúscula e dígito.

  Limitação conhecida do Shell (plus-shell): não há React Router. O shell só
  carrega LoginPage do MFE auth; o link "Criar conta" aponta para /register mas
  o shell não expõe RegisterPage — a URL muda, a UI de login permanece. Para
  registar, usar curl no endpoint /auth/register.

5.1 Permissões por role (admin / vendedor)

  Operação                          | admin | vendedor
  ----------------------------------|-------|----------
  Criar SALE                        |  sim  |  sim
  Criar PURCHASE                    |  sim  |  não
  Listar / consultar                |  sim  |  sim
  Reservar grade                    |  sim  |  sim
  Confirmar / completar SALE        |  sim  |  sim
  Confirmar / completar PURCHASE    |  sim  |  não
  Cancelar qualquer pedido elegível |  sim  |  não
  Cancelar pedido próprio           |  sim  |  sim (createdBy = sub do token)

  cd plus-infra
  cp .env.example .env
  make setup

  URLs após subida da stack:
    Shell         http://localhost:3000
    MS Auth       http://localhost:3001
    MFE Auth      http://localhost:4001
    MS Pedidos    http://localhost:3007/docs
    MFE Pedidos   http://localhost:4007

  Notas operacionais:
    - Na primeira execução, tentativas de import Terraform podem falhar se os
      recursos ainda não existirem no Ministack; o apply subsequente cria-os.
    - Se plus-ms-auth reiniciar em loop, atualizar terraform/rds.env e recriar
      o contentor (scripts/write-rds-env.ps1 no Windows).
    - Reset completo: make reset ou docker compose down -v seguido de make setup.
    - O tsconfig do plus-mfe-ped não deve referenciar tipos de teste (vitest,
      jest-dom) sem as dependências correspondentes instaladas.
    - plus-ms-ped precisa do mesmo JWT_SECRET que plus-ms-auth (variável em
      plus-infra/.env; docker-compose passa-a para o contentor pedidos).
    - Após alterar JWT_SECRET ou rds-ped.env: docker compose up -d --force-recreate plus-ms-ped
    - Após alterar código do plus-ms-ped, rebuild da imagem Docker é obrigatório:
      cd plus-infra && make rebuild-ms-ped
      (equivalente a docker compose build + force-recreate)
      Se endpoints responderem not_implemented, a imagem está desatualizada.

8.1 Como obter token e testar POST /orders

  Pré-requisito: stack a correr (make setup) com plus-ms-auth e plus-ms-ped.

  Opção A — login no Shell (http://localhost:3000):
    email admindev@admin.com, senha Senha123 (utilizador da seed).
    O token fica em localStorage plus.auth.token na origem :3000.
    Para curl, copiar o valor ou usar a opção B.

  Opção B — login via curl (recomendado para testar a API de pedidos):

    curl -s -X POST http://localhost:3001/auth/login ^
      -H "Content-Type: application/json" ^
      -d "{\"email\":\"admindev@admin.com\",\"password\":\"Senha123\"}"

    (Em bash/Linux, trocar ^ por \ e ajustar aspas.)

    Da resposta JSON, usar o campo access_token.

  Opção C — registar novo utilizador (role vendedor) e depois login:

    curl -s -X POST http://localhost:3001/auth/register ^
      -H "Content-Type: application/json" ^
      -d "{\"email\":\"vendedor.teste@loja.com\",\"password\":\"Senha123\"}"

  Criar pedido (admin ou vendedor em SALE; só admin em PURCHASE):

    curl -s -X POST http://localhost:3007/orders ^
      -H "Authorization: Bearer SEU_ACCESS_TOKEN" ^
      -H "Content-Type: application/json" ^
      -d "{\"type\":\"SALE\",\"items\":[{\"productVariantId\":\"var-vestido-marinho-48\",\"quantity\":1}],\"notes\":\"Venda teste\"}"

  Resposta esperada com imagem Docker atualizada: HTTP 201 e JSON com id, status DRAFT,
  createdBy = email do token.

  Cenários esperados:
    - Sem token → 401
    - JWT_SECRET diferente entre auth e pedidos → 401 ou 500 no pedidos
    - Imagem Docker antiga (sem rebuild) → 501 not_implemented em POST /orders
    - vendedor + type PURCHASE → 403
    - admin + type PURCHASE com payload válido → 201
    - items vazio → 400
